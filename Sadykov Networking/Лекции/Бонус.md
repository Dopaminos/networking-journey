# Как проходит curl запрос
Допустим, мы curl-им google.com.

## Проверка URL

curl разбирает url, определяя протокол подключения к нему, hostname и порт.

пример: https, google.com, 443 (HTTPS)

## DNS resolve

Чтобы подключиться к сайту, хостовому устройству не всегда важен **домен**, ему важен **адрес**, отвечающий на вопрос `куда?`.

curl проверяет hosts лист, DNS кэш и Конфигурацию resolver-а, после чего выполняется запрос к днсу (A-запись):

```
ДНС, я не нашел в локальном кэше IP-адрес этого домена.
Помоги мне его найти.
```

И далее, DNS возвращает нам адрес сайта, с которым curl и работает.

## Routing decision

Система смотрит таблицу маршрутизации, чтобы узнать, куда нужно направить трафик. 

Так как google находится в интернете и не локален, в таблице не будет нужного адреса, который вернул DNS, из-за чего трафик направится в Default Gateway.

## ARP

Теперь нужен MAC-адрес этого шлюза, и если его не найти, ARP кидает шлюзу запрос и он возвращает нужный MAC, после чего ARP кэш обновляется.

## TCP (L4)

Создаётся TCP-соединение имеющее в себе три статуса:
- SYN
- SYN-ACK
- ACK
После получения третьего статуса, мы переходим на следующий уровень
## TLS handshake (L5-L7)

Для нашего примера TLS используется для шифрования трафика, проверки сертификатов и обмена ключами между узлами связи. 

## HTTP Request

И наконец, финал.
curl отправляет реквест гуглу, и если сервер отдаёт 200 (OK) - возвращает нам страничку гугла


# Типовые ошибки curl

## URL

Если URL битый, https имеет в себе ошибку или протокол подключения не поддерживается - запрос даже не пройдёт

## DNS Resolve

`curl: (6) Could not resolve host` обычно значит, что нет доступа к DNS серверу, /etc/resolv.conf неправильно настроен или есть какая-то опечатка.

### Как проверить:
```
dig google.com (подача запроса днс-у на получение адреса гугла)
cat /etc/resolv.conf (проверить конфиг, вдруг ошибка какая)
ping ip.of.your.dns (проверка доступности DNS сервера)
```

## Routing / Reachability

Если DNS дал IP, но путь недоступен, то из-за неправильной маршрутизации или связности curl выдаст `curl: (7) Failed to connect` или timeout.

Обычно это значит, что Default Gateway неверно настроен, Default Route-а нет или firewall режет доступ

### Как проверить:
```
ip route (получим таблицу)
ip route get ip.of.your.dns (получим пусть до днс сервера)
ping gateway (проверим доступность шлюза)
traceroute google.com (посмотрим, на каком хопе не получается достичь гугла и искать проблему на последнем доступном IP)
```

## ARP / L2 Gateway Access

Если MAC шлюза получить невозможно, то пакет не уйдёт, даже имея правильный маршрут. Обычно, ошибками в таком случае являются Host Unreachable или Timeout

### Как проверить
```
ip neigh (показать ARP таблицу)
arping <gateway_ip> (пингануть шлюз по ARP)
tcpdump -i eth0 arp (проверим, что говорит интерфейс при подключении к ARP)
```

## TCP 

Если IP найден и маршрут есть, то curl пытается установить TCP соединение, если установить его не удалось, то выдаётся ошибка `curl: (7) Failed to connect to host`.

Причины просты, закрытый порт, сервис не слушает порт, firewall режет TCP или сервер недоступен.

### Как проверить 
```
ss -tulpen (show stat tcp/udp pid extended numeric)
nc -vz google.com 443 (netcan verbose zero i/o, установление соединения без передачи данных и закрытие соединения в случае успеха)
tcpdump -i eth0 tcp ((проверим, что говорит интерфейс при подключении по TCP)
```
